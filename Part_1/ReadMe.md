### Постановка задачи
В данной части был реализован *64-битный потокобезопасный счетчик*, состоящий из двух *32-х битных счетчиков*. Таким образ, можно представить счетчик `counter = [high_ ; low_]`. При увеличении значения счетчика, рано или поздно произойдет ситуация, когда low_ достигнет максимума => со следующим увеличением, необходимо будет очистить low_ и повысить high_ на единицу. Получается, необходимо согласовать работу n потоков при работе с счетчиком и обеспечить отсутсвие разрывов и дубликатов, не используя алгоритм блокировки (lock-free).

Для работы с счетчиком создадим класс counter и определим в нем два основных метода - reset() и fetch_add.
1. reset(value) - сборос счетчика до указанного значение;
2. fetch_add() - атомарное увеличение счетчика на 1 и возврат предыдущего значения.

```C++
class counter {
 public:
  void reset(uint64_t value = 0);
  uint64_t fetch_add();
};
```

### Получение значение и атомарное увеличение на единицу
В методе `fetch_add()` мы должны прочитать текущее 64-х битное состояние счетчика. Чтобы избежать блокировок, применим эвристический метод чтения с проверкой переполнения.
1. Читаем старшую часть `crt_high`, затем младшую `crt_low`, НО! В это время другой поток мог уже изменить состояние нашего счетчика -> `high_` мог измениться. Чтобы избежать такой ситуации введлем проверку `check`, которую сравним с `crt_high`.
2. Если `crt_high == check`, значит за время чтения младшей части переноса разряда не произошло.
3. Если `crt_high != check`, значит другой поток успел переполнить `low_` и увеличить `high_`. Тогда, нам нужно решить, к какому из значений `high_` относится прочитанный нами `crt_low`:
- Если `crt_low < 0x80000000`, значит `crt_low` находится в начале диапазона. Это значит, что мы прочитали `low_` уже после того, как оно обнулилось. Следовательно, оно относится к обновленному значению старшей части — к `check`.
- Если `crt_low >= 0x80000000`, значит `crt_low` велико. Мы успели прочитать `low_` еще до того, как оно переполнилось, но помним, что `high_` обновилось сразу после этого. А значит, `crt_low` относится к старому значению старшей части — к `crt_high`.

```C++
uint32_t actual_high;
if (crt_high == check) {
  actual_high = crt_high;
} else {
  actual_high = (crt_low < 0x80000000 ? check : crt_high);
}
```

### CAS-цикл
После установки корректности полей, мы сформировали 64-х битный результат
```C++
uint64_t crt_value = (static_cast<uint64_t>(actual_high) << 32) | crt_low;
```
Теперь мы можем приступить к увеличению счетчика -> обновлению полей.
1. Увеличиваем младшую часть `upd_low = crt_low + 1`
2. Выполняем обмен через `compare_exchange_weak`:
- Пытаемся атомарно заменить `low_` на `upd_low`
- Если за время вычислений другой поток успел изменить `low_`, CAS вернет `false`, и наш поток уйдет на следующую итерацию цикла
- Если `upd_low == 0`, мы зафиксировали факт переполнения 32-битного счетчика. В этом случае текущий поток берет на себя ответственность за инкремент старшей части через `high_.fetch_add(1, ...)`

### Объяснение используемых моделй памяти
1. Так как предполагается, что метод `reset(value)` выполняется эксклюзивно, то для операций записи используется флаг `std::memory_order_relaxed`, так как они не требуют синхронизации.
2. Метод `fetch_add()` считаем критической секцией, поэтому внутри нее необходимо синхронизовать операции чтения/записи.
- При *чтении* устанавливаем флаг `std::memory_order_acquire` - таким образом гарантируем, что любые изменения, сделанные другими потоками (через `std::memory_order_release`), будут видимы нашему потоку *_до того_*, как мы начнем вычисления.
- При *модификации* устанавливаем флаг `std::memory_order_acq_rel` - таким образом гарантируем, что чтение выполняется *немедленно* (*данные актуальны*), а результат модификации публикуется *незамедлительно*.
- При *увеличении high_* устанавливаем флаг `std::memory_order_release` - таким образом гарантируем, что все потоки, читающие `high_` через `std::memory_order_acquire`, прочитают *актуальные* данные (см. пункт "При *чтении*").
