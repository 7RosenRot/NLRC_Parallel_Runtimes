### Структура счетчика:
Предусмотрено два класса счетчиков: для 1 компонента и для 8 - 2 компонент. *Многокомпонентный* класс counter использует *однокомпонентный* класс.

### Логика работы счетчика:
1. Фактически, по сравнению с захардокоженым двукомпонентным счетчиком изменилось немногое. Нам все также нужно следить за *границами* составных счетчиков и ловить момент смены *старшего бита*.
Главное отличие - теперь мы проходимся *рекурсивно* по каждой из компонент.
2. За рекурсию отвечает метод `add()` многокомпонентного класса counter. Допустим, у нас есть счетчик состоящий из `<uint16_t, uint32_t, uint16_t>`. Мы начинаем с нижнего значения *(крайнее левое)* и идем вправо.
- Мы создаем поля нашего класса, где `low_` отвечает за текущее значение *(uint16_t)*, а `high_` включает в себя все оставшиеся компоненты *(uint32_t, uint16_t)*.
- В следующей итерации поля станут: `low_` *(uint32_t)*, `high_` *(uint16_t)* - теперь мы можем взаимодействовать с `high_` как с "базовым случаем".
3. Сам алгоритм обработки границ не изменился - если мы видим, что страший бит младшего счетчика изменился (0 -> 1 / 1 -> 0) - инкрементируем старший счетчик
```C++
if (old_low >> shift_ != crt_low >> shift_) {
  high_.add();
}
```
4. Для синхронизации используем ту же идею: старший бит младшего счетчика совпадает с младшим битом старшего счетчика, если нет - инкрементируем старший счетчик
```C++
crt_high = crt_high + ((old_low >> shift_) != (crt_high & 1ULL));
```
- В этой реализации использовать предыдущую логику с *вычитанием*: `(old_low >> shift_) - (crt_high & 1ULL)` не получится, потому что таким образом мы корректируем занчение только на границе двух счетчиков, а если старший счетчик уже инкрементирован, переполнился и инкрементировал следующий счетчик, то в попытке догнать его, мы только откатимся назад *(получим -1)*
